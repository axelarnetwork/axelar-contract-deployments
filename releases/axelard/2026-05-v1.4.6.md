## Background

`v1.4.6` is the consensus-breaking upgrade that follows the `v1.3.x` line. It is the first public release on the `v1.4.x` line; earlier `v1.4.0` – `v1.4.5` tags were internal-only pre-releases and should not be deployed.

### Changes since v1.3.11

#### State Machine Breaking

- **Remove link-deposit protocol.** [#2321](https://github.com/axelarnetwork/axelar-core/pull/2321) Deletes the legacy non-GMP token-transfer paths (linked deposit addresses, direct token transfers) that were marked deprecated in `v1.3.1`. Pending link-deposit state is cleaned up in the upgrade handler and the corresponding message types are removed from the API.
- **Reduce `maxBitmapSize` in nexus maintainer state from 32,768 to 1,024.** Lowers per-validator memory pressure during message routing.
- **Bump CosmWasm `wasmd` from v0.54.3 to v0.54.7.** Pulls upstream fixes; no contract-side migration required.

#### Bug Fixes

- Reject votes on failed polls (avoids spurious slashing risk).
- (rosetta) Handle `MsgSendOperation` with fee operations.

#### Improvements

- Update CometBFT to v0.38.23.

### CosmWasm Contracts

No Amplifier contract migrations are required for this upgrade.

## Deployment Steps for Axelar Maintainers

1. Submit and pass the upgrade proposal. Plan name **`v1.4`**.

    ```bash
    axelard tx gov submit-proposal \
        software-upgrade "v1.4" \
        --upgrade-height $upgradeHeight \
        --deposit ${govProposalDepositAmount}uaxl \
        --description "This proposal is intended to upgrade axelar core to v1.4" \
        --title "Axelar v1.4" \
        --from validator --gas auto --gas-adjustment 1.2

    axelard tx gov vote ${proposalId} yes \
        --from validator --gas auto --gas-adjustment 1.5
    ```

2. Inform validators about the upgrade and upgrade height.
3. Wait for the proposal to pass.
4. When the upgrade height is reached, follow the steps for validators below.

### Post-Upgrade Checklist for Maintainers

- [ ] All validator nodes are producing blocks on `v1.4.6`
- [ ] No live signing/voting sessions are stuck after the height transition
- [ ] GMP smoke test: source-chain → Axelar → destination-chain message
- [ ] IBC smoke test: transfer between Axelar and a connected Cosmos chain
- [ ] Multisig key rotation smoke test (if a rotation is scheduled near the upgrade)
- [ ] Confirm no client is still attempting link-deposit messages (these are now rejected)

## Deployment Steps for Axelar Validators

### Prerequisites

- **Building from source:** Go 1.25. Recommended: `make build-static` for a statically linked binary that does not need `libwasmvm` on the host.
- **Latest prebuilt binaries:** [release page](https://github.com/axelarnetwork/axelar-core/releases/tag/v1.4.6). For Linux, prefer `axelard-linux-amd64-v1.4.6-static`.
- **Dynamic binary (fallback):** requires `libwasmvm v2.2.4` ([libwasmvm.x86_64.so](https://github.com/CosmWasm/wasmvm/releases/download/v2.2.4/libwasmvm.x86_64.so)) on `LD_LIBRARY_PATH`. Same version as v1.3.x — no change required if already running the dynamic v1.3.x binary.
- Memory: keep the **64 GB RAM** floor from the v1.3.0 migration. v1.4.6 itself does not trigger a long migration, but the headroom remains required for healthy block processing.

### Upgrade Process

1. Wait for the upgrade height to be reached. The node will halt with a log like:
    ```
    UPGRADE "v1.4" NEEDED at height: <upgradeHeight>
    ```
2. Stop `axelard`, `vald`, and `ampd` (if you run a verifier).
3. Replace the binaries:
   - `axelard` → `v1.4.6`
   - `vald` → `v1.4.6` (bundled in the same `axelard` build)
   - `tofnd` → `v1.0.1` (no change vs. v1.3.x)
4. No `app.toml` / `config.toml` changes are required relative to a node already running `v1.3.x` with the post-v1.3.0 settings (`iavl-cache-size = 0`, `iavl-disable-fastnode = true`, `timeout_commit = 1s`).
5. Restart `axelard`, `vald`, and `ampd`.
6. (Verifiers only) `ampd` version compatibility: confirm against the corresponding `ampd` release note in `releases/ampd/`.

### Post-Deployment Checklist

- [ ] Node produces blocks on `v1.4.6`
- [ ] `axelard version` reports `1.4.6`
- [ ] `axelard status` reports `catching_up: false` once synced
- [ ] (Validators) `vald` resumes voting / signing
- [ ] Test [GMP calls](../evm/EVM-GMP-Release-Template.md)

## Troubleshooting

### `app hash` mismatch

Make sure all replicas across your fleet were swapped to `v1.4.6` before the upgrade height. Mixed binaries across replicas after the upgrade height will fork.

### `libwasmvm` errors with the dynamic binary

Same as `v1.3.x`. Use `libwasmvm v2.2.4` and ensure `LD_LIBRARY_PATH` contains its directory, or switch to `axelard-linux-amd64-v1.4.6-static`.

### Rollback

This is a state-machine-breaking upgrade. **There is no in-place rollback** to `v1.3.x` once a node has executed the `v1.4` upgrade handler — the link-deposit removal and parameter changes are persisted to state. Operators must restore from a pre-upgrade snapshot if rollback is required.

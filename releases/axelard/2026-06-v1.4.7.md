## Background

`v1.4.7` is the recommended binary for the consensus-breaking `v1.4` upgrade that follows the `v1.3.x` line. `v1.4.6` was the first public release on the `v1.4.x` line; `v1.4.7` is a follow-up patch on top of it and is the version that should be deployed. Earlier `v1.4.0` to `v1.4.5` tags were internal-only pre-releases and should not be deployed.

### Changes since v1.3.11

- **Removes the link-deposit protocol** ([#2321](https://github.com/axelarnetwork/axelar-core/pull/2321)). The legacy non-GMP token-transfer paths (linked deposit addresses, direct token transfers) that were marked deprecated in `v1.3.1` are now fully removed. Pending link-deposit state is cleaned up in the upgrade handler and the corresponding message types are removed from the API.
- **Reduces `maxBitmapSize` in the nexus maintainer state from 32,768 to 1,024.** Lowers per-validator memory pressure during message routing.
- **Bumps CosmWasm `wasmd` from v0.54.3 to v0.54.7.** Pulls upstream fixes; no contract-side migration required.
- Updates CometBFT to v0.38.23.
- Rejects votes on failed polls (avoids spurious slashing risk).
- (rosetta) Handles `MsgSendOperation` with fee operations.
- Restores the `InterfaceRegistry` registrations for tss `UpdateParams` and `HeartBeat` ([#59](https://github.com/axelarnetwork/axelar-core/pull/59)). This is the only change in `v1.4.7` relative to `v1.4.6`.

## Migrate CosmWasm Contracts

No Amplifier contract migrations are required for this upgrade.

## Deployment Steps for Axelar Maintainers

1. Submit upgrade proposal and vote on it. Plan name is `v1.4`:

    ```bash
    axelard tx gov submit-proposal \
        software-upgrade "v1.4" \
        --upgrade-height $upgradeHeight \
        --deposit ${govProposalDepositAmount}uaxl \
        --description "This proposal is intended to upgrade axelar core to v1.4" \
        --title "Axelar v1.4" \
        --from validator --gas auto --gas-adjustment 1.2

    axelard tx gov vote ${proposalId} yes \
        --from validator --gas auto --gas-adjustment 1.5
    ```

2. Inform all validators about the upgrade and upgrade height.
3. Wait for proposal to pass.
4. When upgrade height is reached, follow the steps for Axelar Validators below to upgrade our nodes.

### Post-Upgrade Checklist for Maintainers

- [ ] All validator nodes are producing blocks on `v1.4.7`
- [ ] No live signing/voting sessions are stuck after the height transition
- [ ] Testing
    - [ ] General testing: GMP calls, IBC transfers, key rotation
    - [ ] Confirm no client is still attempting link-deposit messages (these are now rejected)

## Deployment Steps for Axelar Validators

### Prerequisites

- **Building from source:** If building from source, use `Go 1.25`. The recommended approach is static linking with `make build-static`, which eliminates additional libwasmvm handling. As a fallback, if you need a dynamically linked binary using `make build`, make sure to update libwasmvm to `v2.2.4` (unchanged since `v1.3.x`).
- **Latest Prebuilt binaries:** Available on the [release page](https://github.com/axelarnetwork/axelar-core/releases/tag/v1.4.7). For Linux, we recommend the statically linked binary [axelard-linux-amd64-v1.4.7-static](https://github.com/axelarnetwork/axelar-core/releases/download/v1.4.7/axelard-linux-amd64-v1.4.7-static) which runs on any Linux distribution without requiring libwasmvm to be installed on your system.

### Upgrade Process

1. Wait for upgrade height to be reached.
2. Stop the validator `axelard` node, `vald`, and `ampd` (if you are running a verifier).
3. Replace the binaries:
   - `axelard` → `v1.4.7`
   - `vald` → `v1.4.7` (bundled in the same `axelard` build)
   - `tofnd` → `v1.0.1` (no change vs. v1.3.x)
4. No `app.toml` / `config.toml` changes are required relative to a node already running `v1.3.x` with the post-v1.3.0 settings (`iavl-cache-size = 0`, `iavl-disable-fastnode = true`, `timeout_commit = 1s`).
5. Restart the validator node, `vald`, and `ampd` with the new binaries. When restarting `axelard` do not do a rollback.
6. Please also upgrade any other Axelar nodes you operate (RPC nodes etc.), and restart your IBC relayer (e.g. hermes).

### Post-Deployment Checklist

- [ ] Verify that nodes are producing new blocks after the upgrade.
- [ ] Verify vald is participating in message validation, signing, heartbeats etc.
- [ ] Test [GMP calls](../evm/EVM-GMP-Release-Template.md)

## Troubleshooting

### `app hash` mismatch

Make sure all replicas across your fleet were swapped to `v1.4.7` before the upgrade height. Mixed binaries across replicas after the upgrade height will fork.

### `libwasmvm` errors with the dynamic binary

Same `libwasmvm v2.2.4` as `v1.3.x`. Ensure system environment variable `LD_LIBRARY_PATH` is specified to its correct location if using the dynamic binary. For docker image no change is necessary.

If you hit the error: `undefined symbol: migrate_with_info`, switch to using the statically linked binary ([axelard-linux-amd64-v1.4.7-static](https://github.com/axelarnetwork/axelar-core/releases/download/v1.4.7/axelard-linux-amd64-v1.4.7-static)). If you need to use a dynamically linked binary, make sure you have the correct `libwasmvm` version (`2.2.4`).

### Rollback

This is a state-machine-breaking upgrade. **There is no in-place rollback** to `v1.3.x` once a node has executed the `v1.4` upgrade handler. The link-deposit removal and parameter changes are persisted to state. Operators must restore from a pre-upgrade snapshot if rollback is required.

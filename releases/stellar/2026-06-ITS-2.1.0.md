## Background

This is the v2.1.0 Stellar ITS release which adds a per-token **flow limiter role**.

The flow limiter role enables:

- The contract operator can delegate flow-limit management on specific tokens to designated addresses via `add_flow_limiter`, `remove_flow_limiter`, and `transfer_flow_limiter` (all operator-only).
- Multiple flow limiters can be approved per `token_id`; revocation removes only the specified limiter.
- Approved limiters can call `set_flow_limit` for their token alongside the operator.
- `is_flow_limiter(token_id, addr)` view function exposes role membership.

This brings Stellar ITS in line with the EVM TokenManager's flow-limiter delegation model.

> **Breaking change (on-chain ABI)**: `set_flow_limit` now requires a leading `caller: Address` argument. The caller must authorize via `require_auth` and must be either the contract operator or an approved flow limiter for the given `token_id`. Integrations calling the old `set_flow_limit(token_id, flow_limit)` form must update to `set_flow_limit(caller, token_id, flow_limit)`.

## Deployment

Create an `.env` config. `CHAIN` should be set to `stellar` for mainnet, and `stellar-2026-q1-2` (or current quarterly chain) for testnet.

```yaml
# Change `PRIVATE_KEY` in `.env` to Stellar
PRIVATE_KEY=<stellar_deployer_key>
ENV=<testnet|mainnet>
CHAIN=<stellar-2026-q1-2|stellar>
```

## Checklist

The following checks should be performed after the rollout.

1. Read-only smoke test of a v2.1.0-only entrypoint:

```bash
ts-node stellar/its.js is-flow-limiter \
    0x1111111111111111111111111111111111111111111111111111111111111111 \
    GBSXKMKLLR64BGH4FEM5NUM3PFKFSZRLPO4IKEQMMDO3HJZ6S4IXVLGW
# expect: "Is Flow Limiter: false"
```

2. End-to-end flow-limiter lifecycle. Requires operator signing for the role-management steps:

```bash
TOKEN_ID=0x1111111111111111111111111111111111111111111111111111111111111111

# Generate an ephemeral limiter identity
stellar keys generate flow-limiter-test --network mainnet
LIMITER=$(stellar keys public-key flow-limiter-test)

# Operator grants the role
ts-node stellar/its.js add-flow-limiter $TOKEN_ID $LIMITER
ts-node stellar/its.js is-flow-limiter $TOKEN_ID $LIMITER    # expect: true

# Switch PRIVATE_KEY to the limiter and exercise the new auth path
PRIVATE_KEY=$(stellar keys secret flow-limiter-test) \
  ts-node stellar/its.js set-flow-limit $TOKEN_ID 1000
ts-node stellar/its.js flow-limit $TOKEN_ID                  # expect: 1000

# Clean up (limiter can also clear the limit)
PRIVATE_KEY=$(stellar keys secret flow-limiter-test) \
  ts-node stellar/its.js remove-flow-limit $TOKEN_ID

# Operator revokes the role
ts-node stellar/its.js remove-flow-limiter $TOKEN_ID $LIMITER

# Drop the ephemeral identity
stellar keys rm flow-limiter-test
```

3. Existing interchain-transfer flow still works (no regression). Run any of the standard Stellar↔EVM transfer smoke tests from previous release docs (e.g. [`2026-01-ITS-1.4.1.md`](./2026-01-ITS-1.4.1.md) §Checklist) to confirm.
